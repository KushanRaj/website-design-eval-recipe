Website design evaluation: reward, metrics, and agent behavior

We built a browser-state evaluator for screenshot-to-code website reproduction. The main finding is clear: broad visual similarity is not enough. The reward needs rendered HTML, CSSOM, visual blocks, and state coverage to distinguish faithful reproduction from a plausible page in the same genre.

1. What we built

The evaluator treats the browser as the source of truth. For each oracle capture, it serves the reference and candidate pages, replays the capture state, then extracts screenshots, rendered DOM/outerHTML, CSSOM, visual blocks, and layout boxes from that live page state.

This matters because future candidates may be plain HTML, React, Solid, Tailwind, or a build system. The reward should not care about source shape. It should care about what the browser renders.

Design principle The oracle manifest defines what is fair to score. If a dropdown, filter, tab, or selected state is in the oracle manifest, then we evaluate whether the candidate can reproduce that visible state.

Pipeline

  1. Generate synthetic oracle website.
  2. Capture oracle screenshots and state manifest.
  3. Give only screenshots to the coding agent.
  4. Agent builds candidate website.
  5. Evaluator generates candidate capture plan and replays states.
  6. Reward combines coverage, visual, DOM, CSSOM, VLM, and geometry metrics.

The run was analytically valid: metrics completed for all 12 trials. Some trials hit an old markdown report writer bug after metrics completed; that bug does not affect the score data used here.

2. Reward overview

The scalar reward is a weighted score per manifest capture. Coverage is applied once as the outer multiplier. Unsupported evaluator components are removed from the available denominator; numeric zeros remain real zeros.

ComponentWeightRole
Screenshot size match0.05Cheap rendering/page-length guard.
Rendered HTML0.10Content and DOM structure.
VLM0.20Human-like visual judgement.
Pixel match0.05Global + blockwise pixel fidelity.
Visual block aggregate0.00Reported as evidence; not directly weighted in the current scalar reward.
BBox geometry0.10Block shape and placement, derived from visual-block matching.
CSSOM style0.10Computed style fidelity over matched visual blocks.
DreamSim0.10Broad perceptual similarity.
Gate A capture must pass screenshot size, rendered HTML, and VLM thresholds before the more specific components contribute. This prevents a page with the wrong content or broken rendering from receiving high credit because one visual metric is high.

13. Agent insights and reward implications

What the agent does well

  • Creates plausible page scaffolds and routes.
  • Captures broad art direction: palette, mood, hero/footer, domain feel.
  • Builds recognizable components such as cards, grids, sidebars, dashboards, menus, and editorial sections.

Where it struggles

  • Exact card density, spacing, row/column geometry, and local alignment.
  • Rendered text/content structure and repeated DOM patterns.
  • Stateful captures: filters, accordions, selected nodes, tabs, toggles, compare states.
  • Custom diagrams/assets such as maps, brackets, rosters, botanical drawings, and audio panels.

Reward changes

  • Keep rendered HTML and CSSOM central.
  • Keep visual block, but always report size/text/position/color/block-pixelmatch.
  • Use blockwise pixelmatch, not global pixelmatch, as the local pixel signal.
  • Keep VLM as a sanity signal, paired with deterministic browser-rendered artifacts.
  • Keep DreamSim small: good guard, weak fine-grained reward.
  • Treat state misses as first-class reward events, not evaluator noise.
Final read The reward is directionally working because it does not simply reward broad visual similarity. Weak candidates can look plausible, but final reward drops when rendered HTML, CSSOM, visual block size, bbox geometry, or state coverage fail.
